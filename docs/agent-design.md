# AIPPT 大纲生成 Agent -- 完整设计方案

> 版本：v1.0 | 日期：2026-03-04

---

## 一、产品定位与核心价值

**一句话定位**：一个能"读懂你想说什么"的 PPT 大纲生成 Agent，将用户的模糊想法或零散素材，转化为逻辑清晰、可直接用于制作 PPT 的完整大纲。

**核心价值主张**：
- 不只是"帮你写大纲"，而是"帮你想清楚你要讲什么"
- 根据输入质量自适应处理：一句话能扩写，散乱文档能重组，结构化文档能还原
- 输出不是干巴巴的标题列表，而是包含故事脉络、设计思路和完整内容的"可执行大纲"

## 二、整体架构设计

### 2.1 系统架构总览

```
用户输入（文本 / 文档 / 文本+文档）
         |
         v
  +-------------------------------+
  |   [M0] 输入预处理模块          |   解析文档、提取文本、规范化输入
  +-------------------------------+
         |
         v
  +-------------------------------+
  |   [M1] 意图分析与场景路由       |   理解用户意图，判断场景类型(a/b/c)
  +-------------------------------+
         |
    +----|----+---------+
    v         v         v
 场景A     场景B     场景C
 结构化    主题扩写   散乱重组
 还原模式  补全模式   整理模式
    |         |         |
    v         v         v
  +-------------------------------+
  |   [M2] 信息补全与研究模块       |   联网搜索、知识补充（主要服务场景B）
  +-------------------------------+
         |
         v
  +-------------------------------+
  |   [M3] 故事线构建模块           |   生成 PPT 的叙事脉络（Storyline）
  +-------------------------------+
         |
         v
  +-------------------------------+
  |   [M4] 大纲细化生成模块         |   输出每页的完整大纲（内容+设计思路）
  +-------------------------------+
         |
         v
  +-------------------------------+
  |   [M5] 质量审查与优化模块       |   检查逻辑连贯性、内容完整性、格式规范
  +-------------------------------+
         |
         v
      最终大纲输出
```

### 2.2 各模块职责详解

#### M0 - 输入预处理模块
| 职责 | 说明 |
|------|------|
| 文档解析 | 支持 PDF、Word、TXT、Markdown 等格式的文本提取 |
| 结构识别 | 检测文档中是否存在明确的分页/分章节标记 |
| 输入规范化 | 将不同来源的输入统一为结构化的中间格式 |
| 元信息提取 | 提取文档标题、作者、日期等元数据 |

**输出格式**：
```json
{
  "raw_text": "用户原始输入文本",
  "documents": [
    {
      "filename": "xxx.pdf",
      "content": "提取的全文",
      "has_page_structure": true,
      "pages": [
        {"page_num": 1, "content": "第一页内容..."},
        {"page_num": 2, "content": "第二页内容..."}
      ],
      "headings": ["标题1", "标题2"],
      "metadata": {"title": "...", "author": "..."}
    }
  ],
  "total_content_length": 5000,
  "input_type": "text_with_document"
}
```

#### M1 - 意图分析与场景路由
| 职责 | 说明 |
|------|------|
| 意图理解 | 分析用户到底想做什么：是汇报、演讲、教学、方案展示？ |
| 场景判断 | 将输入归类到三种场景之一（或混合场景） |
| 参数提取 | 提取 PPT 受众、场合、风格偏好、页数期望等 |
| 澄清决策 | 判断是否需要向用户追问，如果信息充足则直接进入下一步 |

**场景判断逻辑**：
```
IF 文档存在 AND has_page_structure == true:
    -> 场景A：结构化还原模式
ELIF 仅有短文本输入（< 200字）AND 无文档:
    -> 场景B：主题扩写补全模式
ELIF 文档存在 AND has_page_structure == false:
    -> 场景C：散乱信息重组模式
ELIF 文档存在 AND 有文本输入:
    -> 根据文档结构化程度判断 A 或 C，同时参考用户文本意图
```

**意图分析维度**：
- PPT 用途：工作汇报 / 商业提案 / 教学课件 / 演讲分享 / 产品介绍 / 其他
- 目标受众：老板/领导 / 客户 / 同事 / 学生 / 公众
- 风格倾向：正式严谨 / 轻松活泼 / 数据驱动 / 故事驱动
- 篇幅期望：简短(5-10页) / 中等(10-20页) / 详细(20页+)
- 特殊要求：是否需要数据图表、是否需要案例、是否有时间限制

#### M2 - 信息补全与研究模块
| 职责 | 说明 |
|------|------|
| 联网搜索 | 针对用户主题进行 Web 搜索，获取相关信息 |
| 知识补充 | 为薄弱环节补充行业知识、数据、案例 |
| 事实核查 | 对用户提供的关键数据进行基本验证 |
| 素材建议 | 推荐可能用到的图表类型、数据来源 |

> 注意：场景A主要跳过此模块（除非内容明显不足）；场景B重度依赖此模块；场景C选择性使用。

#### M3 - 故事线构建模块
| 职责 | 说明 |
|------|------|
| 叙事结构设计 | 选择合适的 PPT 叙事框架（如：问题-方案-价值、现状-挑战-策略-行动） |
| 逻辑主线构建 | 确定 PPT 从头到尾要讲的"一条线" |
| 节奏把控 | 安排信息密度的节奏感（开头吸引、中间充实、结尾有力） |
| 分页规划 | 确定总页数和每个章节的页数分配 |

**常用叙事框架**：
1. **问题驱动型**：现状 -> 问题/挑战 -> 解决方案 -> 预期效果 -> 行动计划
2. **成果展示型**：背景 -> 目标 -> 做了什么 -> 取得了什么成果 -> 下一步
3. **方案提案型**：市场机会 -> 我们的方案 -> 核心优势 -> 商业模式 -> 合作方式
4. **教学讲解型**：为什么学 -> 是什么 -> 怎么用 -> 实践练习 -> 总结回顾
5. **趋势分析型**：行业背景 -> 关键趋势 -> 影响分析 -> 应对策略 -> 总结展望

**Storyline 输出格式**：
```json
{
  "narrative_framework": "问题驱动型",
  "core_message": "一句话核心信息",
  "storyline": [
    {
      "section": "开篇引入",
      "purpose": "引起注意，建立共鸣",
      "pages": [1, 2],
      "key_point": "行业正在面临XX变革"
    },
    {
      "section": "问题分析",
      "purpose": "深入剖析痛点",
      "pages": [3, 4, 5],
      "key_point": "当前存在三大核心问题"
    }
  ],
  "total_pages": 15,
  "emotional_curve": "低开 -> 制造紧迫感 -> 给出希望 -> 推向高潮 -> 坚定收尾"
}
```

#### M4 - 大纲细化生成模块
| 职责 | 说明 |
|------|------|
| 页面内容填充 | 为每一页生成具体的文字内容 |
| 设计思路标注 | 为每一页建议布局方式和视觉元素 |
| 创作思路说明 | 解释每一页"为什么这样写" |
| 过渡衔接设计 | 确保页与页之间的逻辑流畅 |

**单页大纲输出格式**：
```json
{
  "page_number": 3,
  "belongs_to_section": "问题分析",
  "page_title": "三大核心挑战",
  "creative_intent": "用并列结构呈现三个痛点，让受众产生'确实如此'的共鸣",
  "design_suggestion": {
    "layout": "三栏并列布局",
    "visual_elements": ["图标", "短文案", "数据标注"],
    "color_tone": "偏冷色调，体现问题的严肃性",
    "animation_hint": "三个痛点逐一出现"
  },
  "content": {
    "title": "当前面临的三大核心挑战",
    "subtitle": "基于XX份调研数据的深度分析",
    "body": [
      {
        "point": "挑战一：效率瓶颈",
        "detail": "行业平均处理时间为XX小时，远高于预期目标",
        "data": "效率仅为标杆企业的60%"
      },
      {
        "point": "挑战二：成本攀升",
        "detail": "过去三年运营成本年均增长15%",
        "data": "成本占比从30%上升至42%"
      },
      {
        "point": "挑战三：人才短缺",
        "detail": "核心岗位离职率高达25%",
        "data": "招聘周期平均延长至3个月"
      }
    ],
    "speaker_notes": "这三个挑战并非孤立存在，它们相互关联..."
  },
  "transition_to_next": "面对这些挑战，我们需要一套系统化的解决方案——"
}
```

#### M5 - 质量审查与优化模块
| 职责 | 说明 |
|------|------|
| 逻辑连贯性检查 | 页面之间的逻辑是否通顺，有无跳跃 |
| 内容完整性检查 | 是否覆盖了用户的所有要点，有无遗漏 |
| 信息密度平衡 | 每页信息量是否合理，避免过满或过空 |
| 格式规范检查 | 输出格式是否符合约定的 JSON Schema |
| 用户意图对齐 | 最终产出是否回应了用户最初的需求 |

## 三、核心工作流

### 3.1 完整链路流程

```
[Step 1] 用户输入
   |
   |-- 纯文本？ 文本+文件？ 纯文件？
   v
[Step 2] M0 输入预处理
   |
   |-- 解析文档，提取文本，识别结构
   v
[Step 3] M1 意图分析
   |
   |-- 分析用途、受众、风格
   |-- 判断场景类型 (A/B/C)
   |-- 决定：是否需要追问用户？
   |       |
   |       |-- 需要追问 -> 返回问题给用户 -> 用户回答 -> 回到 Step 3
   |       |-- 信息充足 -> 继续
   v
[Step 4] M2 信息补全（按场景决定深度）
   |
   |-- 场景A: 轻量补全（仅补缺失信息）
   |-- 场景B: 深度搜索补全（联网搜索+知识扩展）
   |-- 场景C: 中度补全（补充逻辑缺失部分）
   v
[Step 5] M3 故事线构建
   |
   |-- 选择叙事框架
   |-- 规划章节与页数
   |-- 输出 Storyline
   |-- -> 可选：让用户确认 Storyline 后再继续
   v
[Step 6] M4 大纲细化生成
   |
   |-- 逐页生成内容+设计思路
   v
[Step 7] M5 质量审查
   |
   |-- 检查逻辑、完整性、格式
   |-- 如有问题 -> 回到 Step 5 或 Step 6 修正
   v
[Step 8] 输出最终大纲
```

### 3.2 追问策略（M1 的关键决策点）

追问的目的是在"不烦用户"和"获取足够信息"之间找平衡。核心原则：

**能推断的不问，必须知道的才问，最多问一轮。**

| 信息类型 | 策略 |
|----------|------|
| PPT 用途 | 优先从内容推断，推断不出才问 |
| 目标受众 | 优先从用途推断，关键场景下询问 |
| 页数期望 | 给出默认推荐，告知用户可调整 |
| 风格偏好 | 不主动问，根据用途和受众自动适配 |
| 核心要传递的信息 | 如果用户输入太模糊（<30字且无文档），必须询问 |

**追问模板示例**：
```
我理解你想做一个关于「{主题}」的 PPT。为了帮你生成更贴合需求的大纲，
想快速确认两个信息：

1. 这个 PPT 主要是给谁看的？（比如：向领导汇报 / 给客户提案 / 团队内部分享）
2. 你最想通过这个 PPT 传达的核心信息是什么？

当然，如果你希望我先出一版，你再调整，也完全可以，直接告诉我"先出一版"就好。
```

## 四、三种场景的具体处理策略

### 4.1 场景A：结构化还原模式

**触发条件**：用户上传的文档中有明确的分页/分章节标记

**核心策略**：尊重用户已有结构，在此基础上优化而非重构

**处理流程**：
1. 识别文档中的分页标记（"第X页"、"Slide X"、Page Break、Heading 层级等）
2. 按用户的分页逻辑拆分内容
3. 对每一页的内容进行"增强"而非"重写"：
   - 补充缺失的标题
   - 优化表述使其更适合 PPT 展示（精简长句、提炼要点）
   - 添加设计建议
   - 补充 speaker notes
4. 构建页面间的过渡逻辑
5. 生成整体 Storyline（从已有结构中提炼，而非重新设计）

**注意点**：
- 保持用户原始的页面数量（除非存在明显的合并/拆分需求，此时给出建议但保留原版）
- 内容措辞可以优化，但核心信息不能丢失或篡改
- 如果用户文档结构已经很好，Agent 的价值在于"锦上添花"：补设计思路、补过渡、补 notes

### 4.2 场景B：主题扩写补全模式

**触发条件**：用户仅输入一句话或短主题，无文档支撑

**核心策略**：像一个经验丰富的 PPT 顾问，帮用户"从0到1"构建内容

**处理流程**：
1. 解析用户输入，提取主题关键词和隐含意图
2. 联网搜索（这是此场景的关键步骤）：
   - 搜索主题相关的行业报告、数据统计
   - 搜索优秀 PPT 案例的结构参考
   - 搜索相关的案例、故事、引用语
3. 基于搜索结果，构建信息框架：
   - 确定要涵盖的子话题
   - 筛选有价值的数据和案例
   - 判断哪些内容适合放进 PPT
4. 选择叙事框架（根据主题类型自动匹配）
5. 生成 Storyline 和详细大纲

**联网搜索策略**：
```
第一轮搜索（广度）：
  - "{主题} PPT 大纲"
  - "{主题} 行业报告 2026"
  - "{主题} 核心趋势/核心问题"

第二轮搜索（深度，基于第一轮结果）：
  - 针对关键子话题的详细数据
  - 具体案例和故事
  - 权威引用和数据来源
```

**注意点**：
- 搜索到的信息需要经过筛选和验证，不能直接堆砌
- 明确标注哪些数据来自搜索（给用户核实的机会）
- 如果主题太宽泛，优先聚焦最有价值的 3-5 个子话题
- 给出的内容要有"观点"，不能只是信息罗列

### 4.3 场景C：散乱信息重组模式

**触发条件**：用户上传了文档但内容缺乏清晰结构

**核心策略**：做"信息的整理师"，帮用户从杂乱素材中提炼出清晰逻辑

**处理流程**：
1. 全量提取文档内容
2. 内容分析与聚类：
   - 识别文档中的关键主题/话题
   - 将散乱的内容按主题归类
   - 识别重复内容并去重
   - 找出内容之间的逻辑关系（因果、并列、递进等）
3. 逻辑重组：
   - 确定最核心的信息是什么
   - 按逻辑关系重新排列内容
   - 识别信息缺口（有论点无论据？有数据无结论？）
4. 适度补全缺失信息
5. 构建叙事结构并生成大纲

**重组策略的关键思路**：
```
Step 1: 提取 -> 从散乱文本中识别所有"信息单元"（一个观点、一个数据、一个案例等）
Step 2: 归类 -> 将信息单元按主题聚类
Step 3: 排序 -> 在每个类别内部按重要性排序
Step 4: 组织 -> 选择叙事框架，将类别映射到 PPT 章节
Step 5: 填补 -> 识别逻辑链上的缺口，补充必要的过渡和衔接内容
```

**注意点**：
- 重组后需要向用户说明"我是这样理解你的材料的"，让用户有机会纠正
- 不能丢弃用户的任何关键信息，即使它看起来"不太重要"
- 如果文档中有明显矛盾的信息，标记出来请用户确认

## 五、关键技术点

### 5.1 文档解析

**技术方案**：

| 格式 | 解析方案 | 说明 |
|------|----------|------|
| PDF | pdf-parse / pdf.js / Claude Vision | 文本型PDF用解析库，扫描型PDF用Claude视觉能力 |
| Word (.docx) | mammoth.js / docx-parser | 提取文本、标题层级、列表结构 |
| Markdown | markdown-it / remark | 直接解析，保留标题层级结构 |
| TXT | 直接读取 | 通过换行、缩进等推断结构 |
| 图片 | Claude Vision | 识别图片中的文字和结构 |

**结构识别规则**（判断是否有明确分页）：
```
强信号（高置信度）：
  - 出现 "第X页"、"Page X"、"Slide X" 等明确标记
  - PDF 的物理分页与内容分段一致
  - Word 文档中使用了分页符

中信号（需要结合上下文判断）：
  - 使用了 H1/H2 标题层级
  - 出现连续编号（1. 2. 3.）且每段内容独立成章

弱信号（倾向判断为无结构）：
  - 仅有段落分隔
  - 编号混乱或不连续
  - 内容主题频繁切换
```

### 5.2 意图分析

**实现方式**：通过 Claude API 的单次调用完成，使用结构化输出。

**关键 Prompt 策略**：
- 提供明确的分类维度和选项
- 要求 Claude 输出置信度，低置信度时触发追问
- 用 few-shot 示例覆盖典型和边缘场景

### 5.3 联网搜索

**推荐方案**：集成 Web Search API（如 Tavily、Serper、Google Custom Search）

**搜索流程设计**：
```
1. 关键词提取 -> 从用户输入中提取搜索关键词
2. 查询构造 -> 生成 2-3 组不同角度的搜索查询
3. 结果获取 -> 调用搜索 API
4. 结果筛选 -> 用 Claude 对搜索结果进行相关性和质量评分
5. 信息整合 -> 将有价值的信息融入大纲生成的上下文
```

### 5.4 大纲生成

**核心技术决策**：采用"分步生成"而非"一次性生成"

原因：
- 一次性生成 15-20 页的详细大纲，质量不稳定
- 分步生成可以让每一步的输出成为下一步的约束，保证一致性
- 更容易做质量控制和错误修正

**生成步骤**：
```
Claude Call 1: 意图分析 + 场景判断 -> 输出分析结果（JSON）
Claude Call 2: Storyline 生成 -> 输入：分析结果 + 用户素材 -> 输出：故事线（JSON）
Claude Call 3: 逐页大纲生成 -> 输入：故事线 + 用户素材 -> 输出：完整大纲（JSON）
Claude Call 4: 质量审查 -> 输入：完整大纲 + 用户原始输入 -> 输出：修改建议或确认通过
```

## 七、潜在挑战与应对方案

### 7.1 技术层面

| 挑战 | 风险等级 | 应对方案 |
|------|----------|----------|
| **长文档处理** | 高 | Claude API 有 context window 限制。对超长文档采用"摘要+关键段落"策略：先让 Claude 对全文做摘要提取，再将摘要 + 关键原文段落作为上下文 |
| **文档解析质量** | 中 | 复杂排版的 PDF/Word 可能解析失败。方案：提供多种解析路径（文本解析 + Claude Vision 兜底），解析失败时告知用户并请求更简单格式 |
| **联网搜索结果质量** | 中 | 搜索可能返回低质量或过时信息。方案：对搜索结果增加 Claude 质量评审环节，标注数据来源和时效性 |
| **生成一致性** | 中 | 多次 Claude 调用之间可能出现前后矛盾。方案：每次调用都传入前序输出作为上下文约束，在 M5 做一致性检查 |
| **响应延迟** | 中 | 多步调用 + 联网搜索导致总耗时较长。方案：能并行的步骤并行执行；在等待时给用户进度反馈；场景A可跳过搜索步骤加速 |

### 7.2 产品层面

| 挑战 | 风险等级 | 应对方案 |
|------|----------|----------|
| **用户期望管理** | 高 | 用户可能期望"完美大纲"一步到位。方案：明确定位为"初稿"，鼓励用户反馈迭代；在 Storyline 阶段设置"确认点" |
| **不同行业的专业度** | 中 | 医疗、法律等行业的 PPT 可能需要专业知识。方案：在联网搜索中增加行业关键词；在输出中标注"建议由专业人士审核" |
| **过度追问导致体验下降** | 中 | 追问太多会让用户失去耐心。方案：严格执行"最多问一轮、最多2个问题"的原则；提供"先出一版再调整"的选项 |
| **大纲与最终 PPT 的鸿沟** | 低 | 大纲再好，做成 PPT 还有很大工作量。方案：大纲中的设计建议尽可能具体可操作；未来可对接 PPT 模板引擎实现自动化 |

### 7.3 边界场景处理

| 场景 | 处理方式 |
|------|----------|
| 用户输入极少（如"做个PPT"） | 触发追问，但提供"热门主题"建议供用户选择 |
| 用户上传多个文档 | 先分析各文档的关系（互补还是独立），再决定合并策略 |
| 文档语言与用户输入语言不一致 | 以用户输入语言为准输出大纲，文档内容翻译后使用 |
| 用户上传了非文本文件（图片/表格截图） | 用 Claude Vision 识别内容，纳入素材 |
| 用户要求的主题敏感或不当 | 礼貌拒绝，说明原因 |

### 8.3 分阶段实施路径

#### Phase 1：核心链路打通（MVP，建议1-2周）

**目标**：实现"文本输入 -> 大纲输出"的最简链路

**范围**：
- M0：仅支持纯文本输入（暂不支持文档上传）
- M1：基础意图分析（默认走场景B）
- M2：暂不实现联网搜索，用 Claude 自身知识补全
- M3：故事线构建
- M4：大纲生成
- M5：暂不实现，人工检查质量

**验证指标**：
- 输入一个主题，能在 30 秒内输出完整大纲
- 大纲结构逻辑清晰，内容可直接用于 PPT 制作
- 试用 10 个不同主题，7 个以上产出质量达标

#### Phase 2：文档能力 + 场景分流（建议第3-4周）

**目标**：支持文档上传，实现三种场景的差异化处理

**范围**：
- M0：支持 PDF、Word、Markdown 解析
- M1：完整的场景判断逻辑和追问策略
- 场景 A、B、C 的差异化处理链路
- M5：基础质量审查

**验证指标**：
- 三种场景各测试 5 个案例，场景判断准确率 > 90%
- 场景 A 的分页还原准确率 > 85%
- 场景 C 的重组后逻辑评分 > 用户原始素材

#### Phase 3：搜索增强 + 质量提升（建议第5-6周）

**目标**：接入联网搜索，大幅提升场景B的内容质量

**范围**：
- M2：接入 Tavily/Serper 联网搜索
- 搜索结果的筛选和整合
- M5：完整的质量审查流程
- 支持用户对 Storyline 的确认和修改

**验证指标**：
- 场景 B 的内容丰富度和准确性显著提升
- 用户满意度评分 > 4/5

#### Phase 4：体验优化 + 迭代能力（建议第7-8周）

**目标**：提供流畅的用户体验，支持多轮迭代

**范围**：
- 流式输出（Streaming）：生成过程实时展示
- 大纲编辑和再生成能力
- 历史记录和模板功能
- 性能优化（并行调用、缓存等）

## 十、总结

这个 Agent 的设计核心思想是 **"以用户意图为中心的自适应大纲生成"**。

三个关键的设计决策：

1. **场景自适应**：不是一套流程走天下，而是根据输入质量和结构自动切换处理策略，让每种输入都能得到最合适的处理方式。

2. **分步生成、逐步约束**：通过 Storyline -> 逐页大纲 的两级生成，先确定"骨架"再填充"血肉"，保证整体逻辑连贯性。

3. **输出不只是"内容"，还有"思路"**：每一页不仅告诉用户"写什么"，还告诉用户"为什么这样写"和"建议怎么设计"，让大纲真正可执行。

建议从 Phase 1 开始快速搭建最小可用版本，尽早让真实用户试用并收集反馈，再逐步完善文档解析和联网搜索能力。
