# Phase 4：从大纲到真实 PPT 预览 -- 完整产品方案

> 版本：v1.0 | 日期：2026-03-07

---

## 一、需求分析

### 1.1 核心价值

当前产品停留在"大纲"层面 -- 用户看到的是结构化的文本卡片，而不是一份看起来像 PPT 的东西。这就好比一个菜谱 App 只告诉你"番茄炒蛋需要番茄和鸡蛋"，却不给你看成品照片。用户脑中无法建立"最终成品"的直觉。

**升级的核心价值是：让用户在生成的那一刻就能看到"成品感"，而不是"半成品"。**

具体来说：
- **降低想象成本**：用户不需要脑补"三栏并列"布局长什么样，直接看到
- **提升信任感**：一个能渲染出真实 PPT 页面的产品，比只输出文本大纲的产品高一个维度
- **提供即时价值**：预览本身就可以截图、演示，不依赖其他工具
- **为后续功能铺路**：编辑、导出、模板市场等高价值功能的前提

### 1.2 用户的真实期望

用户说"生成 PPT"时，实际期望的是：

| 期望层级 | 具体内容 | 优先级 |
|---------|---------|--------|
| **必须有** | 16:9 的页面比例、合理的排版、色彩和字体 | P0 |
| **应该有** | 不同页面布局有视觉差异（标题页 vs 内容页 vs 数据页） | P0 |
| **最好有** | 类似 PPT 的翻页浏览体验 | P1 |
| **惊喜** | 页面中有占位图形、图表、图标等视觉元素 | P2 |

**关键洞察**：用户不需要像素级还原 PowerPoint 的效果，而是需要"看起来像一份正式 PPT"的视觉呈现。这个"像"的关键要素是：固定比例的画布 + 合理的排版 + 统一的色彩系统 + 布局多样性。

### 1.3 当前数据结构的支撑能力评估

好消息是，M4 输出的大纲数据已经包含了非常丰富的设计信息：

| 数据字段 | 渲染用途 | 支撑程度 |
|---------|---------|---------|
| `designSystem.palette` | 页面配色 | 充分 -- 有具体色值 |
| `designSystem.styleTone` | 整体风格判断 | 充分 |
| `designSystem.typography` | 字体选择 | 部分 -- 是描述性文字，需映射 |
| `design.layout` | 页面布局模板选择 | 充分 -- 有明确的布局名称 |
| `design.visualElements` | 装饰性视觉元素 | 部分 -- 是描述性文字，需转为占位 |
| `content.body[].type` | 内容块渲染方式 | 充分 -- 5种类型可映射 |
| `content.headline/subheadline` | 标题渲染 | 充分 |

**结论**：现有数据结构足以支撑一个视觉效果不错的 PPT 预览，不需要修改 M4 的输出格式。

## 三、技术方案 -- 大纲数据到 HTML 页面的转换

### 3.1 Layout 映射策略

M4 输出的 `design.layout` 字段值是自然语言描述，需要映射为具体的 CSS 布局方案。

**核心思路：定义 8-10 种布局模板组件，通过关键词匹配选择模板。**

| Layout 关键词 | 布局模板 | CSS 策略 |
|--------------|---------|---------|
| `全屏标题页` / `封面` | `CoverLayout` | flex 居中，大字标题 + 副标题，背景色填充 |
| `左图右文` | `ImageTextLayout` | grid 两栏 40/60 分割，左侧占位图区 |
| `右图左文` | `TextImageLayout` | 同上，方向翻转 |
| `三栏并列` | `ThreeColumnLayout` | grid 三等分，每栏一个要点卡片 |
| `上下分割` | `TopBottomLayout` | grid 两行，上方标题区，下方内容区 |
| `全图背景+文字叠加` | `FullImageLayout` | 渐变背景 + 居中白色文字 |
| `数据图表+解读` | `ChartLayout` | 左侧大面积图表占位 + 右侧解读文字 |
| `时间轴` | `TimelineLayout` | 横向/纵向时间轴，节点排列 |
| `对比分析` | `ComparisonLayout` | 两栏 VS 对比，中间分隔线 |
| `要点列表` | `BulletListLayout` | 标题 + 纵向列表，每项带图标 |
| 默认/未匹配 | `DefaultLayout` | 标题 + 内容块纵向排列 |

**映射函数示意：**

```typescript
function resolveLayout(layoutStr: string): LayoutType {
  const normalized = layoutStr.toLowerCase();
  if (/标题页|封面|首页/.test(normalized)) return 'cover';
  if (/左图右文/.test(normalized)) return 'image-text';
  if (/右图左文/.test(normalized)) return 'text-image';
  if (/三栏|三列/.test(normalized)) return 'three-column';
  if (/上下/.test(normalized)) return 'top-bottom';
  if (/全图|背景/.test(normalized)) return 'full-image';
  if (/图表|数据/.test(normalized)) return 'chart';
  if (/时间轴/.test(normalized)) return 'timeline';
  if (/对比/.test(normalized)) return 'comparison';
  if (/要点|列表/.test(normalized)) return 'bullet-list';
  if (/结尾|致谢|谢谢|感谢/.test(normalized)) return 'ending';
  return 'default';
}
```

### 3.2 设计系统应用策略

M4 输出的 `designSystem` 包含色板、字体风格、视觉风格等信息，需要转化为可用的 CSS 变量。

**色板映射：**

```typescript
// designSystem.palette 示例: ["#0F172A", "#2563EB", "#E2E8F0"]
// 映射为 CSS 变量：
// --slide-primary:   palette[0]  // 主色，用于标题、强调
// --slide-accent:    palette[1]  // 强调色，用于按钮、高亮
// --slide-bg:        palette[2]  // 背景色/浅色底
// --slide-text:      根据主色自动计算对比色
```

对于色板中颜色数量不足的情况，自动补充：
- 只有 1 个颜色：将其作为主色，自动生成浅色版和深色版
- 有 2 个颜色：第一个为主色，第二个为强调色，自动生成背景色
- 有 3+ 颜色：按顺序分配

**字体映射：**

```typescript
function resolveFont(typography: string): string {
  if (/衬线|宋体|serif/i.test(typography)) return '"Noto Serif SC", serif';
  if (/圆体|可爱/i.test(typography)) return '"Noto Sans SC", sans-serif';
  // 默认无衬线现代字体
  return '"Inter", "Noto Sans SC", system-ui, sans-serif';
}
```

### 3.3 ContentBlock 渲染策略

不同 `type` 的 ContentBlock 在 PPT 页面中的渲染方式：

| type | 渲染方式 |
|------|---------|
| `point` | 带图标的要点卡片，标题加粗，detail 正文。用于列表、网格展示 |
| `quote` | 引用样式：大引号装饰 + 斜体文字 + 来源署名。独占一行，居中 |
| `data` | 数据高亮卡片：数字放大加粗 + 描述文字。适合数据驱动型页面 |
| `imageSuggestion` | 占位图区域：渐变背景 + 图标 + 描述文字。表明此处应有图片 |
| `chartSuggestion` | 占位图表：用 CSS 绘制简化的图表轮廓 + 描述文字 |

### 3.4 imageSuggestion / chartSuggestion 的处理

这是没有真实素材时最需要巧妙处理的部分。

**方案：生成有品味的占位视觉元素**

```
imageSuggestion:
+----------------------------------+
|                                  |
|     ┌──────────────────────┐     |
|     │   渐变背景 + 图标      │     |
|     │   "AI 技术演进图"      │     |
|     │   ---- 配图建议 ----   │     |
|     └──────────────────────┘     |
|                                  |
+----------------------------------+

chartSuggestion:
+----------------------------------+
|     ┌──────────────────────┐     |
|     │  ██                  │     |
|     │  ██  ██              │     |  ← 用 CSS 绘制简化柱状图
|     │  ██  ██  ██          │     |
|     │  ██  ██  ██  ██      │     |
|     │  "市场规模趋势图"      │     |
|     └──────────────────────┘     |
+----------------------------------+
```

具体实现：
- 使用 `designSystem.palette` 中的颜色生成渐变背景
- 图表用 CSS 绘制简化的柱状图/折线图轮廓（不需要真实数据，只是视觉暗示）
- 显示 `detail` 字段的文字描述
- 整体风格统一、有质感，不能让占位区域显得廉价

## 四、技术方案对比

### 方案 A：前端纯模板渲染

**思路**：在前端用 React 组件 + Tailwind CSS，将大纲数据直接映射为 HTML 页面。预定义 10+ 种布局模板组件，根据 `design.layout` 选择对应模板。

**架构：**
```
PPTOutline 数据 → resolveLayout() → 选择 Layout 组件 → 注入色板/字体 → 渲染 HTML
```

**优势：**
- 渲染速度极快（纯前端，无需额外 API 调用）
- 开发成本可控（复用现有 Tailwind 体系）
- 一致性好（模板经过精心设计，质量稳定）
- 不增加 LLM token 消耗
- 实时预览（流式生成一页就能渲染一页）

**劣势：**
- 布局多样性受限于预定义模板数量
- 视觉创意有天花板（模板化意味着规范化）
- 对于"全图背景"等需要真实图片的布局，占位效果可能不够理想

**开发成本：** 中等（约 3-5 天）

### 方案 B：LLM 生成 HTML

**思路**：新增一个 M5 模块，让 LLM 为每一页生成定制的 HTML/CSS 代码。

**架构：**
```
PPTOutline 数据 → M5 Prompt → LLM → 每页 HTML/CSS 字符串 → iframe / dangerouslySetInnerHTML 渲染
```

**优势：**
- 每页布局独一无二，创意上限高
- 可以根据内容动态调整视觉表现
- 理论上视觉效果更好

**劣势：**
- 生成时间大幅增加（12 页 PPT 需要 12 次额外 LLM 调用，至少多 30-60 秒）
- token 消耗翻倍甚至更多
- 质量不稳定（LLM 生成的 CSS 可能有兼容性问题、溢出、对齐偏差）
- 安全风险（需要沙箱渲染用户生成的 HTML）
- 调试困难（出了视觉问题难以定位）
- 流式预览体验差（必须等每页 HTML 完整生成后才能渲染）

**开发成本：** 高（约 7-10 天，包含 Prompt 调优和沙箱机制）

### 方案 C：混合方案

**思路**：前端模板渲染为主，对于关键的 1-2 页（如封面页）使用 LLM 生成定制 HTML。

**架构：**
```
PPTOutline 数据 → 大部分页面走模板渲染
                → 封面页等特殊页面走 M5 LLM 生成
```

**优势：**
- 兼顾速度和创意
- 关键页面效果好

**劣势：**
- 架构复杂度增加
- 两种渲染方式的视觉风格可能不一致
- 维护成本高

**开发成本：** 高（约 6-8 天）

### 对比总结

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 渲染速度 | 即时 | 慢 30-60s | 慢 5-10s |
| 视觉质量 | 中上（稳定） | 高（不稳定） | 中上 |
| Token 消耗 | 0 | 高 | 低 |
| 开发成本 | 3-5 天 | 7-10 天 | 6-8 天 |
| 维护成本 | 低 | 高 | 中 |
| 流式预览 | 支持 | 不支持 | 部分支持 |
| 风格一致性 | 高 | 低 | 中 |

## 五、推荐方案：方案 A（前端纯模板渲染）

### 推荐理由

1. **用户体验优先**：即时渲染 > 等待 LLM 生成。用户已经等了 15-30 秒生成大纲，不应该再等 30-60 秒渲染页面。方案 A 在大纲生成完成的瞬间就能展示 PPT 预览。

2. **质量可控**：模板经过精心设计和测试，每一种布局都经过视觉审查，不会出现 LLM 生成的 CSS 溢出、对齐偏差等随机问题。

3. **成本效率**：零额外 token 消耗，对用户（API Key 费用）和产品（性能）都友好。

4. **渐进增强路径清晰**：先用模板打好基础，后续可以逐步引入 LLM 增强（如为占位图生成 SVG 插画），不需要推翻重来。

5. **技术栈统一**：完全复用现有的 React + Tailwind 体系，不引入新的安全风险（iframe 沙箱）或复杂度。

6. **与现有流式架构兼容**：M4 流式输出一页，前端立即渲染一页的 PPT 预览，体验丝滑。

### 方案 A 的品味保障

模板化不等于粗糙。通过以下策略确保视觉品质：

- **精心的排版**：严格遵循排版原则 -- 留白充足、对齐精确、层次分明
- **动态色彩**：每份 PPT 使用 designSystem.palette 生成独特配色，不千篇一律
- **布局多样**：10+ 种布局模板，覆盖常见 PPT 页面类型
- **细节打磨**：装饰线条、渐变、阴影、圆角等细节让页面有质感
- **占位图的品味**：占位区域使用渐变 + 图标 + 描述文字，而不是灰色方块

## 六、MVP 范围定义

### 6.1 MVP 做什么（P0）

| 功能 | 描述 |
|------|------|
| PPT 页面渲染引擎 | 将 Page 数据渲染为 16:9 的 HTML 页面 |
| 8 种核心布局模板 | cover, bullet-list, image-text, three-column, chart, timeline, comparison, ending |
| 设计系统应用 | 从 designSystem 提取色板、字体，应用到所有页面 |
| 幻灯片浏览模式 | 单页展示 + 左右翻页 + 页码指示 |
| 缩略图总览模式 | 网格展示所有页面缩略图 |
| 视图切换 | 大纲视图 / PPT 预览视图的 Tab 切换 |
| ContentBlock 渲染 | 5 种类型的差异化渲染 |
| 流式预览 | 大纲生成过程中已完成页面可预览 |

### 6.2 MVP 不做什么

| 功能 | 推迟原因 |
|------|---------|
| 页面内容编辑 | 增加大量交互复杂度，Phase 5 再做 |
| 导出为 .pptx | 用户明确说不需要 |
| 真实图片/图表 | 需要接入图片 API / 图表库，Phase 5 再做 |
| 动画/过渡效果 | 锦上添花，不是核心需求 |
| 自定义模板 | 需要模板系统，Phase 6 再做 |
| 演讲者模式 | 需要复杂的多窗口逻辑，后续做 |
| 全屏演示模式 | 可以后续快速添加，MVP 不阻塞 |

### T1: 幻灯片渲染基础设施（0.5 天）

**目标**：创建 PPT 页面的基础容器组件，实现 16:9 画布和缩放逻辑。

**交付物：**
- `src/components/slides/SlideFrame.tsx` -- 16:9 画布容器
- 实现画布自适应缩放（容器宽度变化时保持 16:9 比例）
- CSS 变量注入机制（色板、字体通过 CSS 自定义属性传入）

**关键实现细节：**
```
SlideFrame 组件
├── 外层容器：overflow-hidden, 保持 16:9 (padding-bottom: 56.25% 或 aspect-ratio: 16/9)
├── 内部画布：固定尺寸 960x540（参考尺寸），用 transform: scale() 缩放适配容器
└── CSS 变量：--slide-primary, --slide-accent, --slide-bg, --slide-text, --slide-font
```

### T2: 布局模板组件库（2 天）

**目标**：实现 8-10 种布局模板，覆盖常见 PPT 页面类型。

**交付物：**
- `src/components/slides/layouts/CoverLayout.tsx` -- 封面/标题页
- `src/components/slides/layouts/BulletListLayout.tsx` -- 要点列表页

### T3: 设计系统引擎（0.5 天）

**目标**：将 M4 输出的 designSystem 转换为可用的 CSS 变量和渲染参数。

**交付物：**
- `src/lib/slide-theme.ts` -- 主题解析引擎

**核心功能：**
```typescript
interface SlideTheme {
  primaryColor: string;      // 主色
  accentColor: string;       // 强调色
  backgroundColor: string;   // 背景色
  textColor: string;         // 文字色
  textColorLight: string;    // 浅色文字
  fontFamily: string;        // 字体族
  headingWeight: number;     // 标题字重
  borderRadius: string;      // 圆角大小
}

function buildSlideTheme(designSystem: GlobalDesignSystem): SlideTheme;
function themeToCSS(theme: SlideTheme): Record<string, string>;  // CSS 变量对象
```

**色彩智能处理：**
- 自动检测色板中颜色的亮暗，选择合适的文字对比色
- 如果色板只有文字描述（如"蓝色系"），映射为具体色值
- 生成渐变色用于装饰元素

**关键视觉设计：**

ImagePlaceholder:

## 九、成功指标

### 定性指标
- 用户看到 PPT 预览时能说出"这看起来像一份真的 PPT"
- 不同主题生成的 PPT 在色彩上有明显差异，不千篇一律
- 封面页、内容页、结尾页有明显的视觉区分

### 定量指标
- PPT 预览渲染时间 < 100ms（大纲数据就绪后到页面呈现）
- 8 种布局模板覆盖 90%+ 的 M4 输出 layout 类型
- 页面内容无溢出、截断的异常比例 < 5%
- 移动端可正常查看（虽然体验降级）

## 十、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| 布局模板无法覆盖 M4 生成的某些 layout 名称 | 中 | 低 | DefaultLayout 作为兜底；持续扩展映射关键词 |
| 长文本溢出画布 | 高 | 中 | 设置最大行数 + 文字截断 + 自适应字号 |
| designSystem.palette 包含无效色值 | 低 | 中 | 增加色值校验，无效时回退到默认色板 |
| 中文字体在不同系统渲染不一致 | 中 | 低 | 使用系统字体栈兜底，不依赖特定字体 |
| 页面过多（20+）时缩略图性能问题 | 低 | 低 | 缩略图使用固定尺寸渲染，不实时缩放 |
